# Git 完整操作

## 角色定位

你是一位严谨的"代码管理员"，负责确保代码在构建通过后安全地提交、合并和推送，保证团队成员都能拉到最新且可用的代码。

## 核心原则（必须遵守）

1. **构建先行**：任何 git 操作之前，必须先执行构建检查（按 `daimagoujian` skill 规范），确保代码能编译通过
2. **差异驱动注释**：通过分析 `git diff` 结果自动生成有意义的中文提交注释，不允许写"修复 bug"这类空泛注释
3. **安全合并**：合并前必须拉取最新代码，发现冲突必须解决后再继续
4. **不丢代码**：任何操作都不能导致自己或他人的代码丢失
5. **中文回复**：始终使用中文回复
6. **不输出MD文档**：不需要输出任何 markdown 文档
7. **操作透明**：每一步 git 操作都要告知用户在做什么、为什么做

## 前置依赖

- **构建检查 skill**：`.cursor/skills/daimagoujian/SKILL.md` — 在 git 操作前必须先执行构建检查流程

## 完整操作流程

### 第一步：构建检查（必须最先执行）

> ⚠️ 此步骤必须在任何 git 操作之前完成

1. **按照 `daimagoujian` skill 执行完整构建检查**
   ```bash
   npx tsc --noEmit 2>&1 | head -200
   ```
2. 如果构建有错误，必须先修复所有构建错误
3. 构建通过后，输出确认信息，再进入 git 操作阶段：
   ```
   ✅ 构建检查通过，开始执行 Git 操作
   ```

### 第二步：当前状态分析

1. **查看当前分支和状态**
   ```bash
   git branch -a
   git status
   ```
2. **查看当前分支与目标分支（通常是 main）的差异**
   ```bash
   git diff main...HEAD --stat
   git diff main...HEAD
   ```
3. **查看最近提交历史**
   ```bash
   git log --oneline -20
   git log main..HEAD --oneline
   ```

### 第三步：差异分析与提交注释生成

1. **分析暂存区和工作区的变更**
   ```bash
   git diff --stat
   git diff --cached --stat
   git diff
   ```
2. **按变更内容分类**：
   - 新增功能（新文件、新组件、新接口）
   - 功能优化（修改已有文件的逻辑）
   - Bug 修复（修复错误行为）
   - 样式调整（UI/CSS 变更）
   - 重构（代码结构调整，不改功能）
   - 配置变更（依赖、配置文件修改）
3. **自动生成提交注释**，格式要求：
   ```
   [类型] 简要描述（一行）

   变更详情：
   - 文件1：具体改了什么
   - 文件2：具体改了什么
   ...
   ```
   类型前缀：
   - `feat:` 新增功能
   - `fix:` 修复 Bug
   - `style:` 样式调整
   - `refactor:` 重构
   - `chore:` 配置/杂项
   - `perf:` 性能优化
   - `docs:` 文档更新

### 第四步：暂存与提交

1. **暂存变更**
   ```bash
   git add -A
   ```
2. **提交代码**（使用第三步生成的注释）
   ```bash
   git commit -m "生成的提交注释"
   ```
3. 确认提交成功
   ```bash
   git log --oneline -3
   ```

### 第五步：拉取远程更新并合并

1. **先切到 main 分支，拉取最新代码**
   ```bash
   git checkout main
   git pull origin main
   ```
2. **切回当前开发分支**
   ```bash
   git checkout <开发分支名>
   ```
3. **将 main 最新代码合并到当前分支**（确保自己分支包含最新的 main 代码）
   ```bash
   git merge main
   ```
4. **如果有冲突，执行冲突解决流程**（见下方"冲突解决"章节）

### 第六步：合并到 main 分支

1. **确保当前分支已包含 main 的最新代码（第五步已完成）**
2. **切到 main 分支**
   ```bash
   git checkout main
   ```
3. **将开发分支合并到 main**
   ```bash
   git merge <开发分支名>
   ```
4. **如果有冲突，执行冲突解决流程**
5. **确认合并成功**
   ```bash
   git log --oneline -5
   ```

### 第七步：推送到远程

1. **推送 main 分支到远程**
   ```bash
   git push origin main
   ```
2. **确认推送成功**
   ```bash
   git log --oneline origin/main -3
   ```
3. **输出最终确认**
   ```
   ✅ 代码已成功推送到远程 main 分支，其他同事现在可以拉取最新代码
   ```

## 冲突解决流程

当 merge 出现冲突时：

### 1. 查看冲突文件
```bash
git diff --name-only --diff-filter=U
```

### 2. 逐个分析冲突
```bash
# 查看具体冲突内容
git diff <冲突文件>
```

### 3. 解决冲突原则
- **保留双方有效代码**：不能丢弃任何一方的有效改动
- **功能完整性**：解决后的代码必须同时包含双方的功能改动
- **语法正确**：确保解决后的代码语法正确，删除所有冲突标记（`<<<<<<<`、`=======`、`>>>>>>>`）
- **构建验证**：冲突解决后再次运行构建检查

### 4. 完成冲突解决
```bash
git add <解决后的文件>
git commit -m "merge: 解决合并冲突 - [冲突描述]"
```

## 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| 构建失败 | 停止 git 操作，先修复构建错误 |
| 推送被拒绝 | `git pull --rebase origin main` 后重试 |
| 合并冲突 | 按冲突解决流程处理 |
| 远程分支不存在 | 检查分支名拼写，必要时 `git fetch --all` |
| 本地有未提交的变更 | 先暂存提交或 `git stash` |

## 关键检查项

- [ ] 构建检查通过（零错误）
- [ ] 提交注释准确描述了本次变更内容(分点)
- [ ] 已拉取远程最新代码
- [ ] 冲突已全部解决（无冲突标记残留）
- [ ] 冲突解决后重新构建检查通过
- [ ] main 分支已推送到远程
- [ ] 推送后确认远程仓库已更新
- [ ] 未丢失任何一方的代码改动

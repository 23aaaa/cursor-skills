# 功能需求开发

## 核心原则（必须遵守）

1. **全面梳理**：详细梳理流程，不限于前端界面、接口、数据库、后端业务，考虑与其他功能的联通
2. **后端优先**：先完成后端修改，再修改前端
3. **代码复用**：修改前系统性检查已有代码和相关代码，寻找可复用的代码资源，减少开发维护成本
4. **精确定位**：不依赖记忆和对文档结构的推断，彻底放弃猜测，用 grep 精确定位，完全匹配所有相关函数和代码，不断重试
5. **最小化修改**：谨慎修改，不影响已有功能
6. **接口稳定**：后端尽量不修改已确定好的接口
7. **避免硬编码**：前端尽量不要硬编码
8. **中文回复**：始终使用中文回复
9. **不输出MD文档**：不需要输出任何 markdown 文档

## 开发流程

### 第一步：需求厘清

1. 仔细阅读用户需求
2. **主动提问**：对不清楚的地方向用户提问，直到完全厘清
3. 梳理需求涉及的模块：数据库、后端接口、前端页面、与其他功能的联通点

### 第二步：代码调研

1. 使用 grep/rg 精确搜索所有相关代码
2. 检查已有的数据库 schema、API 路由、组件、hooks
3. 列出可复用的代码资源
4. 识别需要新增和修改的部分

### 第三步：制定开发计划

使用 TodoWrite 创建开发任务列表，按以下顺序：

```
Task Progress:
- [ ] 数据库 Schema 修改（如需要）
- [ ] 后端业务逻辑开发
- [ ] 后端 API 接口开发
- [ ] 后端接口测试验证
- [ ] 前端组件/页面开发
- [ ] 前后端联调
- [ ] 与其他功能联通检查
```

### 第四步：后端开发（优先）

1. **数据库**：修改 schema，确保字段设计合理
2. **业务逻辑**：实现核心业务逻辑
3. **API 接口**：实现 RESTful API，遵循已有接口规范
4. **验证**：确保后端功能正确

### 第五步：前端开发

详细规范请参考 [references/frontend-patterns.md](references/frontend-patterns.md)

**核心要求**：
- 容器/展示组件分离
- 展示组件纯负责 UI，容器负责逻辑
- 前端无数据处理，只关注 UI（数据处理放后端）
- 避免硬编码
- **遵循设计系统**：先读取 `theme.css` 学习已有的设计变量（颜色、间距、字体、圆角等），所有样式必须使用设计系统中的变量，不允许硬编码颜色值或随意自定义样式

**响应式布局规范（必须遵守）**：
- 任意宽度下（320/375/768/1024）页面不得出现横向溢出或内容超出视口
- 所有横向排列（flex row / grid）都必须可收缩：文本区域必须可截断，按钮/图标区域不得把容器撑宽
- 不允许使用 `100vw` 做主容器宽度；如必须使用，需保证不叠加 padding/margin 导致超宽
- flex/grid 子项：默认给可能放文本的子项加 `min-w-0`；长文本默认加 `break-words` 与截断策略（单行 `truncate`，多行 `line-clamp`）
- 图片/图标必须 `max-w-full`，容器内媒体必须有固定尺寸或 `aspect-ratio`，禁止加载后撑破布局
- 任何绝对定位/transform 不得造成元素布局盒超出视口；如使用必须在容器内裁剪且可滚动

**状态管理选择**：
| 场景 | 方案 |
|------|------|
| 服务端状态（API 数据） | TanStack Query |
| 全局客户端状态（UI 状态） | Zustand |
| 低频全局状态（主题/用户） | React Context |
| 组件局部状态 | useState |

### 第六步：联调与验证

1. 检查前后端接口调用是否匹配
2. 检查路由定义是否正确
3. 检查字段命名是否一致
4. 验证与其他功能的联通点

## 关键检查项

- [ ] 后端接口未破坏已有接口契约
- [ ] 前端无硬编码数据
- [ ] 前端样式使用 theme.css 设计系统变量，未自定义颜色/间距
- [ ] 响应式布局无横向溢出（320-1024 各断点验证）
- [ ] flex/grid 文本子项已加 min-w-0 和截断策略
- [ ] 图片/媒体已限制 max-w-full，不会撑破布局
- [ ] 前端只做 UI 渲染，不做数据处理
- [ ] 数据库 schema 变更向后兼容
- [ ] 所有相关代码已通过 grep 精确定位并检查
